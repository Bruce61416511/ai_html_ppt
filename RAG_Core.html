标准化

通用全局标准：清洗规则 + 切片参数

前置清洗规则

统一编码为 UTF-8
清除页眉、页脚、页码、水印
清除空行、特殊乱码符号、重复分隔符
结构化数据禁止原生格式入库
必须转自然语言 + 维度元数据
所有切片强制携带：文件名 / 页码 / 文件类型 / 解析方式

切片参数（行业最优）

标准文本

chunk_size=600~800, overlap=10%~15%

OCR/ASR 噪声文本

chunk_size=800, overlap=20%

噪声数据加大重叠率，杜绝语义断裂

结构化数据

chunk_size=500~600, overlap=10%

PPT/单页短内容

优先按页切片，不强行拆分

铁律：噪声数据加大重叠率；结构化数据不存原生符号，必须转自然语言

检索方法

更多检索方法：SPLADE、全文检索、稀疏向量

除了 BM25 和 Dense，还有几种值得关注的方法，各有适用场景

SPLADE 稀疏向量检索

介于 BM25 和 Dense 之间。用神经网络学到每个词的重要性权重，生成稀疏向量（大部分维度为0）。既有关键词匹配的精确性，又有语义扩展能力

学出来的关键词权重稀疏向量

适合：需要精确匹配但不满足于 BM25 固定权重的场景

大白话：死记关键词变活 —— AI 自己学会哪些词重要

Elasticsearch 全文检索

ES 原生支持倒排索引全文检索。可以替代 BM25 作为关键词一路，也能存储文本+元数据。部分版本已内置向量检索，一个引擎搞定混合检索

倒排索引全文检索

适合：已有 ES 基础设施、不想额外部署向量库的团队

大白话：不用向量库也行，搜索引擎那套直接拿来搜

ColBERT 延迟交互

不是把整段压缩成一个向量，而是每个 Token 一个向量。检索时问题 Token 和文档 Token 两两算相似度再求和。精度接近 Cross-Encoder，速度接近 Dense

Token级向量

适合：对检索精度要求极高又承担不起 Rerank 全量成本的场景

大白话：不是整句压缩成一个向量，每个词独立比对，像"找茬"一样精细

多模态检索

不仅检索文本。CLIP 等模型把图片也向量化，可以"以文搜图"或"以图搜图"。回到 Part 1 的思路——图片描述文本已经入库，文本检索覆盖了大部分需求

文本+图像

适合：电商以图搜图、设计素材库检索

大白话：不仅搜文字，还能拿张图搜相似的图

方法论：BM25 + Dense 是基线，SPLADE/ColBERT 是增强，ES 是工程选择。根据精度和成本选型
